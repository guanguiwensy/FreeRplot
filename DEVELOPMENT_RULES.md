# Development Rules

> 本文件记录项目持续开发过程中的前提约束。  
> 在开始任何新开发、重构、修 bug、补功能之前，必须先阅读本文件，并结合 `PROJECT_GUIDE.md` 一起执行。

## 1. 目标

本项目是一个 AI + R 的长期演进工具。开发时优先保证：

- 可维护性
- 模块边界清晰
- 低耦合
- 可定位问题
- 可持续扩展

不允许为了短期速度，破坏长期结构。

## 2. 开发前必读

在开始任何开发任务前，默认按以下顺序阅读：

1. `DEVELOPMENT_RULES.md`
2. `PROJECT_GUIDE.md`
3. 本次任务直接相关的模块文件

如果需求与本文件冲突，必须先指出冲突，再决定如何实现。

## 3. 架构拆分原则

当前项目已经进行了 UI 与逻辑拆分，后续必须保持，不允许重新退化成大单文件。

- 聊天 UI、绘图 UI、设置 UI、代码 UI、数据 UI、图库 UI 等子 UI，继续保持分文件组织。
- Server 逻辑继续按模块拆分，如 `mod_ai_chat`、`mod_data`、`mod_plot`、`mod_settings`。
- 新功能优先放入已有模块边界内；如果必须新增模块，应继续沿用当前拆分风格。
- 不要把本应属于子模块的逻辑重新堆回 `ui.R`、`server.R` 或其他总装文件。

## 4. 文件注释规范

每个 `.R` 文件开头都应有清晰的大段注释，后续必须保持这一规范。

- 新建 `.R` 文件时，必须补充文件头注释。
- 大幅修改现有 `.R` 文件时，应同步更新文件头注释。
- 注释至少应说明：文件用途、依赖关系、导出函数、关键数据流或模块职责。
- 注释风格尽量与当前仓库保持一致。

## 5. 依赖与环境隔离原则

依赖隔离是本项目的核心约束之一，后续开发必须严格遵守。

- 除用户明确同意可以作为全局基础依赖的关键 R 包外，其余包和函数都不应随意进入全局作用域。
- 已明确可接受的全局基础依赖包括这类包：`ggplot2`、调色相关包、数据整理相关包，以及用户后续明确批准的少数关键包。
- 其他依赖应尽量限制在模块内部使用，不制造新的全局变量污染。
- 后续新增或重构代码时，应按 `{box}` 的隔离思路组织；即使某些旧代码尚未完全切换，也不能继续扩大全局泄漏面。
- 如果某个实现会引入新的全局符号、跨模块隐式依赖或不透明副作用，应先收紧设计，再实现。

## 6. 调试、日志与报错原则

每个子模块都必须具备可定位的问题出口，报错不能模糊。

- 调试信息要能明确指出所属模块。
- 报错要尽量从对应模块抛出，帮助快速定位 bug。
- 优先沿用现有的 `MODULE <- "..."`、`log_debug()`、`log_info()`、`log_warn()`、`log_error()`、`safe_run()` 模式。
- 用户可见报错应简洁清楚；开发日志应包含足够定位信息。
- 不能出现“出了错但看不出来自哪个模块”的实现。

## 7. 修改代码时的执行准则

- 优先复用现有模块边界、辅助函数和数据流，不随意复制逻辑。
- 保持 `global.R` 中的加载顺序稳定；如确需调整，必须整体审视依赖关系。
- 新增功能时，优先增强现有结构，而不是绕过结构。
- 修 bug 时，优先修正模块内部责任边界，不做跨模块脆弱补丁。
- 如果发现现有实现与本文件不完全一致，后续修改应朝本文件要求收敛，而不是继续偏离。

## 8. 对 AI 助手的约束

后续在本仓库中开发时，默认执行以下规则：

- 开发前先阅读本文件与 `PROJECT_GUIDE.md`。
- 理解相关模块后再动手，不跳过结构分析直接修改。
- 所有实现都默认服从本文件中的模块化、注释化、隔离化、可调试化要求。
- 若用户后续提出新规范，本文件应继续更新，并以最新版本为准。

## 9. 一句话原则

这个项目后续的开发方向不是“先能跑”，而是：

**在保证能跑的前提下，始终优先保证结构清晰、边界明确、依赖可控、问题可定位。**
